4信任链 · Auth谁颁发 · 谁验签

Web 用户视角

浏览器登录 server → session token

⇒ web 调 server: Bearer session_token ✓
⇒ web 调 matter: Bearer session_token
            ↳ matter 共享 JWK 本地验签
              (不调 server, 避免 N+1)
⇒ web 调 runtime: Bearer session_token
            ↳ runtime 同上

Daemon 视角

daemon 启动 → 用 api-key register runtime
                          │
runtime → server: verify api-key ✓
                  (X-Internal-Token)
                          │
runtime 颁发 runtime_token (JWT)
  scope: daemon_id + space_id
  signed by 共享私钥

⇒ daemon 调 matter: Bearer runtime_token
            ↳ matter 用共享公钥本地验签 ✓
⇒ daemon 调 runtime: Bearer runtime_token ✓
JWK 共享的代价 · runtime 是唯一 issuer · server / matter 是 verifier。3 个服务共享一个 public key（pem 文件或 KMS 路径）。零额外网络调用。代价是 key rotation 需要协调 — PoC 期可接受。

6替代设计对比push / outbox 方案为什么没选

维度替代方案 · push / outbox本 plan · daemon pull
触发模式matter outbox → runtime pushdaemon 主动 pull from matter
matter → runtime 调用有 (outbox webhook)砍掉
runtime → matter 调用有 (HMAC writeback)砍掉
runtime → server 调用有 (mint bot OBO + verify api-key)保留 · 不可避免
matter 复杂度+ outbox 表 + outbox worker + HMAC 验签+ bot_task 表 + 1 个 internal endpoint
runtime 复杂度高 (outbox · HMAC · writeback worker)低 (daemon mgr · bot mapping)
daemon 复杂度中 (被推)中高 (主动 pull · 协调两端)
失败爆炸半径matter outbox 卡死 → 全停daemon 挂 → 单机停 · 域更小
时延push 即时pull 间隔 3-10s
